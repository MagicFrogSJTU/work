# 公开benchmark


## 定位

因研发人力有限，公开benchmark不追求“全量复现”，而是做有限复现。

## 目标

- 用统一设置复现少量、公开、知名、权威benchmark
- 验证公开资料中的关键结论是否大体可信
- 建立内部稳定可复用的对比基线

## 选型原则

- 与重点关注方向一致：纯文本、图像理解、Agent能力
- 社区认可度较高，便于向内部解释
- 数据集与评测脚本相对成熟，复现成本可控
- 能覆盖当前纳入评测的主流模型
- 单次测试成本可接受，适合月度持续更新

## 可直接复用的公开资源

- benchmark 数据集
- 官方或社区评测脚本
- 统一评测框架，如 lm-evaluation-harness、OpenCompass
- 推理与部署组件，如 vLLM 等开源服务框架
- 社区已有结果，可作为 sanity check

## 建议测试范围

第一期建议控制在 6 到 8 个 benchmark，先覆盖核心能力，不追求大全。

### 纯文本

- MMLU-Pro：适合做通识与复杂学科理解的横向比较
- GPQA：适合看高难知识问答能力
- BBH：适合看复杂指令与推理稳定性

建议：第一期至少选 2 个，优先 MMLU-Pro + GPQA。

### 图像理解

- MMMU：适合看综合多模态理解能力
- DocVQA：适合看文档理解与 OCR 相关能力
- ChartQA：适合看图表读取与分析能力

建议：第一期至少选 2 个，优先 MMMU + DocVQA。

### Agent能力

- BFCL：适合看函数 / 工具调用的准确性
- GAIA：适合看复杂任务完成能力
- TAU-bench：适合看业务流程类任务执行能力

建议：第一期至少选 2 个，优先 BFCL + GAIA。

## 第一版推荐组合

如果资源有限，先固定以下 6 个 benchmark：

- MMLU-Pro
- GPQA
- MMMU
- DocVQA
- BFCL
- GAIA

这样可以覆盖：
- 文本能力
- 图像理解
- 工具调用与复杂任务执行

如果后续要增强代码或数学评估，再增补 HumanEval、SWE-bench、GSM8K 等专项任务。

## 测试口径建议

为避免结果不可比，公开benchmark复现时应统一：

- 固定模型版本
- 固定 prompt 模板
- 固定采样参数，如 temperature、top_p
- 固定是否允许工具调用
- 固定是否允许多轮重试
- 记录运行日期、评测脚本版本、数据集版本

当前阶段不单独比较开源模型的在线 API 与内网部署差异，先聚焦模型结果本身。

## 使用建议

- 公开benchmark只做“有限复现”，不要把人力花在全量追榜
- 如果某个benchmark与智船真实业务关联很弱，可以降低优先级
- 如果某模型在公开benchmark表现很好，但部署成本过高，仍不应直接进入最终推荐
- 最终结论必须与智船benchmark结合，不单独依赖公开benchmark
